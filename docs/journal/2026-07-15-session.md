## Codex session — debug prediction queued/deadlock

คำขอและอาการจากผู้ใช้:
- ผู้ใช้แจ้งว่า prediction เข้า queued และค้างที่ “รอ worker” พร้อม PostgreSQL `deadlock detected`
- error ระบุ process สองตัวรอ `AccessExclusiveLock` บน relation OID 33676 และ 33693

การวินิจฉัย:
- query `pg_class` ยืนยัน OID 33676 คือ `sim_runs` และ 33693 คือ `debate_posts`
- หน้า Job Center poll `/simruns.json` และ `/run-metrics.json` พร้อมกัน; endpoint ทั้งคู่สร้าง `RunStore` แล้วเรียก `setup()` ทุก request
- `RunStore.setup()` เดิมรัน `CREATE/ALTER TABLE` และ `DROP/ADD sim_runs_status_check` ทุกครั้ง จึงขอ AccessExclusiveLock ซ้ำบนสอง relation และเกิด deadlock เมื่อ requests วิ่งพร้อมกัน
- process inspection พบ FastAPI ทำงาน แต่ไม่มี Celery worker; Redis จึงรับ task ได้แต่ไม่มี consumer และมี retry หัวข้อเดียวกันค้าง 3 งาน (รวม run ที่ผู้ใช้ cancel แล้ว)
- ตรวจเพิ่มพบ `mark_running()` เดิมเปลี่ยนสถานะโดยไม่เช็ก current status ทำให้ stale task ใน Redis สามารถปลุก run ที่ canceled กลับมารันและเสียค่า LLM ได้

การแก้และเหตุผล:
- เพิ่ม process-local `_SETUP_DONE` + thread lock เพื่อให้ polling requests ใน process เดียวไม่แตะ schema ซ้ำ
- เพิ่ม `pg_advisory_xact_lock` ใน transaction เพื่อ serialize schema setup ระหว่าง API กับ Celery ซึ่งเป็นคนละ process
- อ่าน constraint definition ก่อนและทำ `DROP/ADD` เฉพาะ schema เก่าที่ยังไม่มีสถานะ `canceled`
- เปลี่ยน `mark_running()` เป็น atomic `UPDATE ... WHERE status = 'queued' RETURNING` และให้ worker ตอบ 409 เมื่อรับ stale/canceled task
- ลบ task ซ้ำ 2 รายการจาก Redis และ mark queued duplicate เป็น canceled เหลือ run ล่าสุดรายการเดียว เพื่อลดความเสี่ยงยิง LLM ซ้ำ
- เปิด Celery worker แบบ `--pool=solo`; inspect ping ตอบ `celery@Santipong: OK`

ผลตรวจจริง:
- stress `/simruns.json` และ `/run-metrics.json` พร้อมกัน 100 requests หลัง reload: HTTP 200 ครบ 100/100 ไม่มี deadlock/503
- worker รับ task ล่าสุดและเปลี่ยน run ออกจาก queued ได้จริง แต่ BudgetGuard หยุดก่อนยิง LLM เพราะยอดเดือนนี้ `$96.01` สูงกว่าเพดาน `$50.00`; run จบเป็น error พร้อมข้อความตรงไปตรงมา

- เพิ่ม regression tests สำหรับ concurrent setup ทำ schema ครั้งเดียว และ canceled run ไม่กลับเป็น running
