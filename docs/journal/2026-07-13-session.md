## ช่วงที่ 1 — Persona Pack Editor: ปรับกลุ่มประชากร + media diet เองได้

**ผู้ใช้ถาม/ขอ**: "ตอนนี้เข้าใจว่าแต่ละกลุ่มจะมี agent ของตัวเอง เช่น กลุ่มคนสูงอายุ, กลุ่ม Gen Z
คำถามคือฉันสามารถปรับแต่งจำนวนกลุ่มหรือรายละเอียดแต่ละกลุ่มได้มั้ย รวมทั้งปรับแต่ง media diet
ที่มาจาก Internet ได้ด้วยตัวเอง แต่แน่นอนว่าควรมีค่า default ไว้ ... ช่วยออกแบบ UX/UI ให้ใช้งานง่ายและสวยงาม"

**คำอธิบายที่ให้ผู้ใช้**: ฐานมีอยู่แล้ว ~60% จาก Persona Packs (P5-M7) — schema + validate_pack
(PII gate GOV-01, share sum, 2-8 กลุ่ม) + CRUD (ยกเว้น UPDATE) + AI-generate + try-ask + pool preview;
สิ่งที่ขาดคือ "การแก้ไขด้วยมือทั้งหมด" (modal เดิมเป็น AI-only, preview อ่านอย่างเดียว)
และ channel_mix ของ segment คือตัวเดียวกับที่คุม media diet ของโต๊ะข่าวสด (P7) อยู่แล้ว —
แก้ที่จุดเดียวได้ผลทั้งการแพร่ในโลกจำลองและข่าวที่กลุ่มเห็น

**มติผู้ใช้ (ผ่าน AskUserQuestion + approve แผน)**:
1. แก้ไขใน **modal ของ wizard เท่านั้น** (แบบ SwarmSight PersonaPackModal 3 tabs)
2. สำมะโน default = **อ่านอย่างเดียว + ทำสำเนาไปแก้** (กันค่าอ้างอิงเพี้ยนเงียบๆ)
3. **ไม่เอา import/export** ในรอบนี้

**งานที่ทำ**:
- Backend: `PackStore.update` (validate_pack ด่านเดียวกับ create — GOV-01 ทุกทางเข้า;
  ไม่แตะ created_by/created_at = provenance TRUST-06), `PUT /personas/packs/{id}`
  (ลำดับ except สำคัญ: PackValidationError ซึ่งเป็น subclass ของ ValueError ต้องจับก่อน →
  PII/validation = 422, id ไม่พบ = 404), pool.json เพิ่ม `voice_activity`
  (จำเป็นให้สำเนาสำมะโนผ่าน validate)
- UI: rewrite `PersonaPackModal.tsx` เป็น 3 tabs — **เลือก** (การ์ดสำมะโน 🔒 + ปุ่มทำสำเนา /
  การ์ด pack: ใช้ในรัน-แก้ไข-ลบ, ลบ pack ที่เลือกอยู่ = ตกกลับสำมะโน), **แก้ไข** (แถบสัดส่วนรวม
  stacked bar + normalize, การ์ดต่อกลุ่มพับได้: ชื่อ/share/voice/นิสัย 3 ตัว/traits chips +
  **media diet 4 ช่องทาง** พร้อมข้อความชี้ชัดว่า "กำหนดทั้งการแพร่ในโลกจำลอง และข่าวที่กลุ่มนี้เห็น
  จากโต๊ะข่าวสด", ลองถามราย segment, save = auto-normalize เมื่อเพี้ยน >1% พร้อมแจ้ง),
  **ให้ AI ร่าง** (ผลลง draft ในแท็บแก้ไขให้ตรวจก่อนบันทึกเสมอ — ไม่ save ตรง)
- component `Slider` ใหม่ใน ui.tsx (ใช้ซ้ำ ~9 ตัว/กลุ่ม), i18n key `pk_*` ครบชุด (ลบ key เก่า
  ของ preview ที่ไม่ใช้แล้ว 4 ตัว)
- Tests 324→329: update roundtrip (created_by คงเดิม), id ปลอม → ValueError,
  update validate เหมือน create, PUT 200/422 share/422 GOV-01/404, pool มี voice_activity
- Smoke จริงกับ server: สำเนาสำมะโนทั้งชุด (มี prior พิเศษ `sensitivity_awareness`
  ที่ validate ยอมรับเพราะเช็คเฉพาะ 3 key บังคับ) → create/update/delete ผ่าน 200 ครบ

## ช่วงที่ 2 — redesign UX persona packs ตาม feedback ผู้ใช้

**ผู้ใช้ขอ**: (1) "ปรับ UX/UI ให้ดูสากลมืออาชีพขึ้น ... เวลาคลิ๊กไม่ควรเป็น popup จาก window system
ควรเป็น popup เราเอง" (2) ระหว่างทำ ผู้ใช้เสริม: "มีรายการ pack ก็ควรกดลบหรือแก้ไขที่หน้านั้นได้เลย
หมายถึงหน้าแรก"

**ที่ทำ**:
- **ย้ายการจัดการ pack มาหน้า wizard ตรงๆ**: การ์ด pack แต่ละใบมี ✏️ แก้ไข / 🗑 ลบ inline +
  mini stacked bar แสดงสัดส่วนกลุ่ม; การ์ดสำมะโนมี 📋 ทำสำเนา; ปุ่ม "+ สร้าง pack เอง" กับ
  "✨ ให้ AI ร่าง pack" อยู่ใต้ grid — ไม่ต้องเปิด modal เพื่อหาปุ่มอีกต่อไป
- **modal เหลือหน้าที่เดียว = ตัวแก้ไข** (2 แท็บ: แก้ไข / ให้ AI ร่าง) เปิดด้วย intent จาก parent
  (edit pack เดิม / สำเนาสำมะโน / pack เปล่า / AI): header + footer action คงที่ เนื้อหา scroll
  กลาง, Escape ปิดได้, บันทึกแล้วปิด modal และ**เลือก pack นั้นให้อัตโนมัติ**
- **ConfirmDialog ของเราเอง** (`ui.tsx`) แทน `window.confirm` ทุกจุดในระบบ: ลบ pack (หน้า wizard),
  ลบ run (หน้าประวัติ), AI ร่างทับร่างที่แก้ค้าง — มี danger variant (ปุ่มแดง + ไอคอน 🗑),
  ยกเลิกได้ด้วย Escape/คลิกพื้นหลัง
- **Smoke ผ่าน browser จริง**: เปิด wizard → ทำสำเนาสำมะโน (7 กลุ่ม โหลดครบ share 100%) →
  เห็น media diet sliders + traits chips → บันทึก → การ์ดใหม่โผล่พร้อมถูกเลือก → ลบผ่าน
  dialog ของเราเอง 2 รอบ; ลบ pack ทดสอบตกค้างจาก test เก่า 16 ใบ

**เหตุผลที่อธิบาย**: popup ระบบ (window.confirm) ข้ามธีม/ภาษา ควบคุมไม่ได้และดูไม่มืออาชีพ —
dialog ของเราคุมทั้ง copy สองภาษา, ปุ่ม danger, และพฤติกรรมคีย์บอร์ดเอง

## ช่วงที่ 3 — sweep ความสม่ำเสมอ UX ทุกหน้า

**ผู้ใช้ขอ**: "ช่วยเช็คทุกหน้าให้หน่อย มันควรเป็นแบบนี้แบบเดียวกัน ปรับได้เลย แนวทางคือให้ใช้งานง่าย"
— ให้ตรวจทุกหน้าว่าเข้ามาตรฐานใหม่ (dialog ของเราเอง + จัดการ inline) แล้วแก้ตรงจุดที่หลุดได้เลย

**ผลตรวจครบ 10 หน้า**:
- **NewRun / Runs (ประวัติ)** — เข้ามาตรฐานใหม่แล้วจากช่วงที่ 2
- **Watchlist** — ช่องว่างจริงที่ใหญ่สุด: สร้างได้แต่**ลบไม่ได้เลยทั้งระบบ** (backend ก็ไม่มี delete)
  → เพิ่มครบ stack: `WatchlistStore.delete` (DELETE RETURNING, ไม่พบ → ValueError) +
  `DELETE /watchlists/{id}` (404 เมื่อไม่พบ) + ปุ่ม 🗑 ต่อรายการ + ConfirmDialog danger;
  alerts ของรายการหายตาม `ON DELETE CASCADE` ที่ schema มีอยู่แล้ว — **ชอบธรรมเชิง governance**
  เพราะ docstring หัวไฟล์ระบุชัดว่า watchlists/alerts เป็นตาราง operational ไม่ติด append-only
  (ต่างจาก audit/registry); รันที่เคยเกิดยังอยู่ใน audit ตามปกติ + test cascade ใหม่ 1 ข้อ
- **Settings** — ปุ่มล้าง LLM/Tavily key เดิมลบทันทีไม่ถาม → ConfirmDialog danger
  พร้อมอธิบายผล (ระบบจะ fallback ไปใช้ key จาก .env)
- **RunDetail** — share dialog เป็น dialog ของเราอยู่แล้ว ปรับ backdrop/shadow/aria
  ให้ตรงมาตรฐานเดียวกับ ConfirmDialog
- **Calibration** — resolve เป็น irreversible (TRUST-01) แต่มี inline confirm 2 ขั้น +
  ช่องแหล่งอ้างอิงบังคับอยู่แล้ว — **ตั้งใจคงไว้** (เหมาะกว่า dialog เพราะต้องกรอกข้อมูลประกอบ)
- **Gallery / Compare / Insights / Landing** — อ่านอย่างเดียว ไม่มีจุดเสี่ยง

**ยืนยันทั้งระบบ**: grep `window.confirm|alert|prompt` ใน web/src เหลือแค่ comment ใน ui.tsx —
ไม่มี popup ระบบเหลืออยู่เลย; ruff + `npm run build` + `uv run pytest -q` เขียวครบ **330 ข้อ**

**หมายเหตุปิดเครื่อง**: ผู้ใช้แจ้งต้องปิดเครื่องกลางงาน — ปิด state ให้สมบูรณ์ตาม protocol
(STATE.md + journal + commit + push + CI) เพื่อกลับมาต่อได้จากไฟล์ล้วนๆ

## ช่วงที่ 4 — discussion "cap 2-8 มาจากไหน / traits อิงอะไร" → ADR-0009

**ผู้ใช้ถาม**: persona packs ทำไมต้องสร้างเป็นกลุ่ม ทำไมจำกัด 2-8 (มาจากสำมะโนไทยไหม เพิ่มได้ไหม)
และ traits (voice/เกรงใจ/มีม-ประชด/say-do gap) กำหนดมาจากไหน ตรงกับประชากรไทยจริงไหม —
ขอให้วิเคราะห์ + research บนอินเทอร์เน็ต + discuss

**คำอธิบายที่ให้ผู้ใช้ (สรุป discussion)**:
1. *ทำไมเป็นกลุ่ม*: segment เป็นแม่พิมพ์ให้ factory ปั๊ม agent รายตัว (jitter ±0.1);
   แนวทาง persona รายบุคคลจริง (Stanford generative agents แม่น ~85%) ต้องใช้บทสัมภาษณ์คนจริง
   ซึ่งเราไม่มีและขัด GOV-01; โครงสร้างกลุ่มสอดคล้อง GOV-02 aggregate-only โดย construction
2. *cap 2-8*: ตอบตรงๆ ว่าเป็น undocumented design choice (ไม่มี ADR/comment, เลข 8 hardcode
   ซ้ำใน React) แต่บังเอิญสอดคล้อง practice ตลาด (3-7 กลุ่ม optimal) — ข้อจำกัดจริงคือสถิติ:
   กลุ่มเล็กต้องมี agent ≥30 ตัวถึงวัดได้
3. *traits*: concept มีฐานวิชาการจริงทุกตัว (kreng jai — Klausner/discourse analysis 26 ความหมาย;
   say-do gap — social desirability bias แรงกว่าใน collectivist cultures; มีม/ประชด — งานวิจัย
   การแสดงออกการเมืองไทยใต้การเซ็นเซอร์) แต่**ตัวเลข default ยังสังเคราะห์** (provenance ประกาศเอง)
   — ที่พิสูจน์แล้วคือทิศทาง (FAB-01 sign test) ไม่ใช่ขนาด; media diet มีข้อมูลจริงให้ calibrate
   (DataReportal 2025: LINE 78.2%; YouGov: TikTok Gen Z 76% vs Boomers 49%)

**มติผู้ใช้**: ทำครบ 3 ข้อ — (A) ADR + citations (B) ขยาย cap 12 + guard สถิติ (C) calibrate media diet

**งานที่ทำ**:
- `MIN/MAX_SEGMENTS = 2, 12` จุดเดียว + `limits` ใน `/personas/pool.json` → UI เลิก hardcode
  (มี `FALLBACK_PACK_LIMITS` ฝั่ง TS); ⚠️ เตือนกลุ่มที่ share×agents<30 ทั้งใน pack editor
  และ pool panel ของ wizard (เตือน ไม่ block — ผู้ใช้รันเล็กเพื่อทดลองได้)
- calibrate `channel_mix` ทั้ง 7 กลุ่มใน segments.yaml (v0.2-draft) จากสถิติแพลตฟอร์มจริง
  พร้อม provenance ใหม่ (คงคำ "สังเคราะห์" สำหรับ share ที่ยังไม่ calibrate — test_p1m3 เช็คคำนี้);
  จุดเปลี่ยนหลัก: young_urban algo 0.45→0.53, elderly algo 0.10→0.14, gig algo 0.30→0.37
- ลบ `sensitivity_awareness` (prior ตัวที่ 4 ที่ไม่มีอะไรอ่าน — รั่วเข้า pack ผ่านสำเนาสำมะโน)
- `docs/adr/ADR-0009-segment-limits-and-channel-mix-calibration.md` — rationale + citations ครบ
- ของแถม: แก้ copy การ์ดสำมะโน "6 segments" → 7 (stale ตั้งแต่แรก)
- Smoke browser จริง: label เพิ่มกลุ่ม (7/12), ⚠️ ขึ้นครบทุกกลุ่มที่ n=100 และหายหมดที่ n=1,000,
  media diet ใน editor โหลดค่า calibrated, ไม่มี prior ตายในสำเนา; tests 330→331
