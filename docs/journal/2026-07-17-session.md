## Codex — กู้ prediction ที่ค้าง queued และปิด readiness false positive

ผู้ใช้รายงานว่าการสร้างคำทำนายใหม่ค้างที่ `รอ worker`/`queued` และตั้งคำถามอย่างตรงไปตรงมาว่าเหตุใด
function หลักจึงใช้งานไม่ได้. ตรวจ run ล่าสุดพบ `debate-20260717-134255-547389-6dea9287` กับ Celery job
`fe3f5c6f-c43b-4ff8-b018-b6113e35ecab` ค้าง queued จริง. PostgreSQL, Redis และ Neo4j พร้อม แต่ไม่มี
Celery worker process; `celery inspect ping` ไม่พบ node. `/health/deep` ยังตอบ ok เพราะ contract เดิมตรวจเพียง
broker Redis ไม่ได้ตรวจ consumer จึงเป็น readiness false positive และ API persist แถว `รอ worker` ทั้งที่ไม่มี
ผู้ประมวลผล.

### การกู้ระบบสด

- เปิด worker แบบ hidden ด้วย solo pool และ subscribe queue `fabric,debate,maintenance`; ล้าง proxy variables
  ของ process เพื่อไม่ให้เกิด OpenRouter route failure แบบเหตุการณ์ก่อนหน้า
- worker รับทั้ง stale canceled task แบบ idempotent และงานผู้ใช้; Tavily/RSS/OpenRouter ตอบ HTTP 200
- Debate run ของผู้ใช้จบ `complete`, progress 100, message `complete` ในประมาณ 92 วินาที
- หลังโหลดโค้ดแก้ใหม่ `/health/deep` รายงาน postgres/redis/worker/neo4j เป็น ok ครบ
- ทำ live zero-cost Fabric smoke ผ่าน endpoint หน้าเว็บ: HTTP 202, run
  `fabric-20260717-135253-950224-327f390e` จบ complete 100% แล้ว DELETE cleanup สำเร็จ

### การแก้ต้นเหตุ

- ADR-0017 กำหนดว่า broker reachable ไม่เท่ากับ worker ready; worker เขียน Redis heartbeat ทุก 5 วินาที
  พร้อม TTL 20 วินาที และ health แสดง worker แยกจาก Redis
- `/runs/async` ตรวจ heartbeat หลัง governance/idempotency lookup แต่ก่อนสร้าง queued row; ถ้าไม่มี worker
  ตอบ 503 ภาษาไทยและไม่ persist run ปลอม ส่วน replay ของ Idempotency-Key ที่เคยรับแล้วคืน run เดิมได้
- Makefile worker command subscribe queue `fabric,debate,maintenance` ให้ตรง task routing
- เพิ่ม regression tests สำหรับ offline rejection/no persistence, idempotent replay หลัง worker offline และ health
  component. Full verification: 422 tests, Ruff check/format และ migration no-op ผ่าน
