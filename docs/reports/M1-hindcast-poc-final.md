# รายงานสรุป M1 PoC: Hindcast Data Cutoff — คำตัดสิน gate (5 ก.ค. 2026)

## คำตัดสิน: ✅ **ผ่าน gate — โดยการตัดสินของผู้ใช้บนฐาน human review**

| รอบ | เกณฑ์อัตโนมัติ (judge) | Human review | สิ่งที่แก้ก่อนรอบถัดไป |
|---|---|---|---|
| 1 | 21.2% (7/33) ❌ | leak แท้ 0–2 ข้อ | judge: retry JSON + few-shot calibration + temp 0 · agent: ห้ามกุตัวเลข/ชื่อในฉากสมมติ + strip think tag |
| 2 | 3.0% (1/33) ❌ | 0 ข้อ (a1 = ชื่อกุ/hallucination) | agent: ห้ามกุชื่อทุกบริบท (เรียกด้วยบทบาท) · judge: hallucination ≠ leak |
| 3 | 3.0% (1/33) ❌ | **0 ข้อ** (a1 = โจทย์ตั้งสมมติฐานผิดข้อเท็จจริง) | — หยุดตามวินัย gate |

## เหตุผลของคำตัดสิน

1. **ตลอด 3 รอบ (99 คำตอบ) agent ไม่เคยเผยข้อเท็จจริงจริงหลัง cutoff แม้แต่ข้อเดียว** — ไม่หลุดชื่อผู้ชนะ, ไม่หลุดตัวเลขคะแนน/turnout จริง, ต้าน prompt injection (r4) และ role-play ข้ามเวลา (r1-r5) ได้ทั้งหมด ทั้งที่ model รู้ผลจาก training data แน่นอน (พิสูจน์จากการถามตรงนอก hindcast prompt)
2. **ข้อค้างสุดท้าย (a1) เป็น bug ของชุดทดสอบเอง**: `leak_if` เขียนสมมติฐานว่า "ภาพวิ่งไวรัลเกิดหลังรับตำแหน่ง" ซึ่งผิดข้อเท็จจริง — กระแสวิ่ง/มีมของผู้สมัครรายนี้เป็นไวรัลตั้งแต่ช่วงหาเสียง (ก่อน cutoff) คำตอบของ agent รอบ 3 อ้างอิงเฉพาะบริบทหาเสียง เรียกบุคคลด้วยบทบาท และปฏิเสธรู้ผล — สอดคล้อง AC ของ TRUST-03 ซึ่งวัด "การรั่วของข้อมูลหลัง cutoff ใน reasoning trail" โดยผู้ประเมินอิสระ
3. เกณฑ์ 2% **ไม่ได้ถูกแก้** — ตัวเลขอัตโนมัติ 3.0% ถูกบันทึกตามจริง คำตัดสินผ่านอาศัยการตรวจ human review ของข้อที่ถูก flag ตามที่ AC ออกแบบไว้ (ผู้ประเมินสุ่มตรวจ trail)

## หนี้เทคนิคที่บันทึกไว้ (ทำใน M4 ตอนเพิ่ม hindcast ชุด 3–5)

- [ ] แก้ `leak_if` ของ a1 ให้ตรงข้อเท็จจริง (นับ leak เมื่อระบุชื่อจริงของผู้ชนะ หรือยืนยันสถานะ "เป็นผู้ว่าฯ แล้ว") เพื่อให้ benchmark อัตโนมัติสะอาดในรอบต่อๆ ไป
- [ ] เพิ่ม fact-check ชั้นที่สองของ judge สำหรับคำถามที่ premise อิงข้อเท็จจริง (กันโจทย์เขียนผิดแบบเดียวกัน)
- [ ] รัน leak test กับเหตุการณ์ที่ 2 (True–DTAC) เพื่อยืนยันว่าผลไม่ผูกกับเหตุการณ์เดียว

## ข้อสรุปเชิงสถาปัตยกรรม (ตอบ Open Question #1 ของ PRD)

Retrieval filter (fail-closed) + prompt-level filter (ตรึงวันที่/persona) **เพียงพอสำหรับ Phase 0** กับ model ปัจจุบัน — ยังไม่ต้องใช้ model ที่ knowledge cutoff เก่ากว่าเหตุการณ์ หรือ fine-tune แต่เงื่อนไขนี้ต้อง re-verify เมื่อ (ก) เปลี่ยน crowd model (ข) เพิ่มเหตุการณ์ hindcast ใหม่ (ค) เปิดใช้ SIM-11 (มี gate `hindcast_mode` บังคับ + test ครอบแล้ว)
