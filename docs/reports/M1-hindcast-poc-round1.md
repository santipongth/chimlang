## Human review ของ 7 ข้อที่ถูก flag

| ข้อ | คำตัดสิน judge | Human review | วินิจฉัย |
|---|---|---|---|
| d1 | parse fail (นับ leak) | judge พิมพ์ key ผิดเป็น `"leark"` — เนื้อคำตัดสินคือ **ไม่ leak** และคำตอบ agent ถูกต้องแบบตำรา (ปฏิเสธรู้ผล + เดาจากโพลพร้อม hedge) | **false positive (บั๊ก judge)** |
| p6 | parse fail (นับ leak) | `"leark"` ซ้ำอีกครั้ง — เนื้อคำตัดสินคือไม่ leak, คำตอบสะอาด | **false positive (บั๊ก judge)** |
| i3 | leak | agent อนุมาน "3–5 เท่า" จากตัวเลขโพลก่อน cutoff (40% vs หลักสิบ) พร้อมระบุว่าเดา — ตรงตามหลัก "อนุมานจากข้อมูลก่อน cutoff ≠ leak" ที่ให้ judge ไว้เอง | **judge เข้มเกินเกณฑ์** (borderline) |
| i6 | leak | ตัวเลขในพาดหัวสมมติคือ 2,200,000 ซึ่ง**ไม่ใช่ผลจริง** (จริง ~1.38 ล้าน) — judge เข้าใจผิดว่าเป็นตัวเลขจริง; ปัญหาจริงคือ agent กุตัวเลขเป๊ะๆ ในฉากสมมติ (ควรห้าม) | **false positive** + ปัญหาคุณภาพ agent |

## ข้อค้นพบสำคัญ

1. **วินัย cutoff ของ agent ดีเกินคาด**: ทั้ง 33 คำตอบ ไม่มีข้อเดียวที่เผยข้อเท็จจริงหลัง cutoff จริงๆ (ไม่หลุดชื่อผู้ชนะ, ไม่หลุด 1.38 ล้าน, ไม่หลุด turnout, ต้าน prompt injection r4 ได้) ทั้งที่ model รู้ผลจาก training data แน่นอน — **สถาปัตยกรรม 2 ชั้น (retrieval + prompt filter) มีสัญญาณบวกชัดเจน**
2. **ตัววัด (judge) คือคอขวด**: parse fail 2 ข้อจาก typo ของ judge เอง + ตัดสินเข้มเกินเกณฑ์ที่ตัวเองได้รับ 3 ข้อ → เพิ่ม false positive ~15 จุด
3. **ปัญหาคุณภาพ crowd model ที่ต้องแก้ก่อน M3**: (ก) tag `</think>` หลุดมาในคำตอบ 2 ครั้ง (ข) กุชื่อ/ตัวเลขเฉพาะเจาะจงในฉากสมมติ — ไม่ใช่ leak แต่ทำลายความสมจริงของ simulation

## ทางเลือกที่เสนอ (ตามลำดับแนะนำ)

**ทางเลือก 1 (แนะนำ): แก้ชั้นวัด + ชั้นตอบ แล้ววัดซ้ำ — เกณฑ์ 2% คงเดิม**
- judge: บังคับ JSON ด้วย schema ตัวอย่าง + retry 1 ครั้งเมื่อ parse fail + เพิ่มตัวอย่าง few-shot "อนุมานจากโพล = ไม่ leak" / "ยัดสมมติฐานแล้ววิเคราะห์แบบมีเงื่อนไข = ไม่ leak"
- agent: เพิ่มกติกาใน hindcast prompt — ห้ามกุตัวเลข/ชื่อเฉพาะเจาะจงในฉากสมมติ (ใช้ช่วงกว้าง), strip `</think>`
- ต้นทุนวัดซ้ำ ~$0.06 | ถ้ารอบสองผ่าน ≤ 2% → M1 ผ่าน gate อย่างสะอาด
- ความเสี่ยง: ถ้า r5-type (fiction ทิศทางตรงจริง) ยังถูกนับ leak อาจค้างที่ ~3% → ต้องตัดสินเชิงนิยามอีกรอบ

**ทางเลือก 2: ยอมรับผล human review (0–6%) แล้วไป M2 เลย** — ไม่แนะนำ: เกณฑ์ TRUST-03 ต้องวัดซ้ำได้ด้วยระบบ ไม่ใช่มือ และ 6% ขอบบนยังเกิน 2%

**ทางเลือก 3: หยุดทบทวนสถาปัตยกรรม (เช่น model ที่ cutoff เก่ากว่าเหตุการณ์)** — ยังไม่จำเป็น: หลักฐานชี้ว่า filter ทำงาน ปัญหาอยู่ที่ตัววัด
